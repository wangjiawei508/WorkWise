## Context

WorkWise 的 renderer 已有语言设置和翻译函数，但 Electron 原生菜单在应用启动时使用固定英文模板，语言变更也没有触发重建。插件市场新增的 CLI 与推荐 Skill 使用了翻译键，却未进入当前资源字典，因此键名被当作兜底文本展示。更新器具备 GitHub Release 检查和 IPC，但后台提醒与通用设置的入口未形成完整闭环。

## Goals / Non-Goals

**Goals:**

- 以同一份持久化语言设置驱动 renderer、原生应用菜单和托盘菜单，并在语言变更后立即重建。
- 让插件市场所有内置卡片都使用可读、简洁的中英文文案；缺失翻译时使用明确的产品兜底文案而不是内部键。
- 复用现有更新器，实现后台提醒、帮助菜单检查更新和通用设置手动检查。
- 将 README 和产品说明调整为中文优先的 GitHub 产品入口。
- 让最终安装包而非仅开发源码通过依赖安全、ASAR 完整性、托管 CLI 和跨平台构建验证。

**Non-Goals:**

- 不改变 CLI/Skill 的安装协议或放宽 Skill 安全限制。
- 不引入新的更新服务、遥测或强制更新。
- 不修改内部运行时架构。

## Decisions

### 主进程集中构建本地化菜单

新增主进程菜单构建模块，读取规范化的 `zh-CN` / `en-US` 语言值，集中生成应用菜单与托盘文案。renderer 保存语言后通过已有设置变更链路通知主进程重建，而不是让 renderer 拼装原生菜单。这样菜单角色、快捷键和 macOS 平台约定仍由 Electron 管理。

备选方案是完全依赖操作系统 locale，但这会违背用户在 WorkWise 内显式选择语言后的预期。

### 内置市场项目使用稳定标识与翻译字典

内置 CLI/Skill 保留稳定项目 ID，显示名、说明和状态标签统一从完整的中英文资源读取。组件层为每个内置项目提供人类可读的静态兜底值，避免资源加载异常时泄漏翻译键。安装错误先映射为简短用户提示，详情按需展开。

### 三个更新入口复用同一状态机

启动后台检查、帮助菜单“检查更新”和通用设置按钮都调用现有 updater 服务。更新状态通过单一 IPC 事件/查询接口同步到 renderer；后台检查只在发现新版本或发生需要用户处理的错误时提示，不打断无更新场景。

### 中文 README 作为默认入口

根 `README.md` 使用中文，顶部提供英文文档链接。产品介绍拆分为可维护的中文文档，README 聚焦价值、截图、能力、安装、更新、安全与帮助，避免堆叠内部实现细节。

### 发布门禁验证最终产物

生产依赖通过锁文件升级和 `npm audit --omit=dev` 检查消除 high/critical 漏洞。after-pack 不再直接删除 ASAR 索引仍引用的 unpacked 文件；运行时依赖应在打包输入阶段裁剪，或保持 ASAR header 与磁盘内容一致。托管 CLI 的 list/install/update/diagnose/remove IPC 必须注册，并在隔离工具目录中覆盖校验和、回滚、版本诊断和外部应用状态。macOS 双架构及 Windows x64 均由相同源码和锁文件构建。

## Risks / Trade-offs

- [macOS 原生菜单更新时机与窗口生命周期不同步] → 在 app ready 后构建，并对语言变更和窗口重新激活使用同一幂等重建函数。
- [更新检查被多个入口并发触发] → updater 复用单飞请求并广播同一状态，按钮在检查中禁用。
- [翻译键继续遗漏] → 增加中英文资源键集合一致性和内置市场渲染测试。
- [后台网络错误造成打扰] → 自动检查静默处理瞬时网络失败，手动检查才显示明确错误。
- [after-pack 破坏 ASAR unpacked 索引] → 增加最终 ASAR 全量读取校验，禁止打包后删除 header 已登记文件。
- [CLI 上游或 GitHub 暂时不可用] → 端到端网络测试与确定性 mock 测试分层；网络失败不得被误报为 IPC 缺失。
- [本机构建无法代表 Windows] → Windows x64 由 GitHub Actions 独立门禁验证并上传测试产物。

## Migration Plan

无需用户数据迁移。现有语言设置继续使用；缺失语言时按系统语言规范化。发布后通过现有 GitHub Release 更新通道分发，出现问题可回滚菜单模板和设置入口而不影响用户数据。
