# 章节写作（Chapter Writing）

本文档展开 SKILL.md 第 5 步的细节，描述每类章节的写作套路。

## 总原则

**评委不是读者，是打分员。** 他们手里拿评审表，挨着评分项找标书的对应内容。你写每一章时心里都要有一句话：这一章是回应哪条评审项的？评委翻到这里能不能立即判断该给多少分？

**复述—展开—收束。** 每个对应评审项的章节用三段式：

1. **复述**：开头 1-2 句把评审标准/规范要求复述一遍（"针对 XX 要求"、"按照评审标准 X.X 条"）
2. **展开**：方案、做法、图表、数据
3. **收束**：1-2 句明确说"本方案完全满足/超过 XX 要求"

这样评委不用费劲推理，直接对应打分。

**少议论、多结构。** 中文标书的特点是图表密集、列表密集。能用表格的别写段落，能用流程图的别用文字描述。

## 各类章节写法

### 项目概述类（项目背景、目标、范围）

来源：规范书的"项目背景""项目目标""建设范围"段。

写法：

- 不要原样复述招标方文字（会被认为没思考）
- 提炼 3-5 条投标方对项目的理解，用"我们认为本项目……""理解到招标方的核心诉求是……"的口吻
- 加一段"本项目的关键挑战"，列 3-4 条挑战 + 每条挑战的应对思路（这部分专门是给"项目理解"评分项加分）
- 配一张"项目目标分解图"或"项目范围边界图"

### 需求理解与点对点应答

这是中文标书最重要的章节之一。做法：

1. 把规范书里所有显式要求条款抽出来，做成一张表：

   | 序号 | 招标方要求（原文摘录） | 投标方应答 | 满足情况 |
   |------|---------------------|------------|---------|
   | 1 | 系统应支持 1000 并发用户 | 本系统采用 XX 架构，实测支持 2000 并发，详见 §3.4 | 完全满足 |
   | 2 | 提供 7×24 小时服务 | 设立专职运维团队，承诺 7×24 响应，详见 §6.5 | 完全满足 |

2. 满足情况列使用“完全满足 / 满足 / 部分满足 / 不满足”四档并如实填写。出现部分满足或不满足时，必须显式提示偏离风险并交由用户决定是否整改、澄清或放弃投标；严禁把真实偏离改写成“满足”。
3. 表格之外，再用一段文字总结："共 XX 条要求，完全满足 XX 条，满足 XX 条，无偏离项。"

### 总体架构

写法：

- 开头一段总述："本方案采用 XX 分层架构 / 微服务架构 / 数据中台架构……"
- 一张总体架构图（用 mermaid 或描述用户用什么工具画）
- 分层介绍：每一层写什么职责、用什么组件、对应规范书哪些要求
- 一段"架构特点"：列 4-6 条优势（高可用、可扩展、安全、易运维…），每条配一句具体支撑

要点：架构图里出现的每个组件，后面 3.2 模块设计章节都要有对应介绍，不能图里画了文字里没了。

### 功能模块设计

每个模块独立成节，模板：

```
#### 3.2.X {模块名}

**模块定位**
（一句话说明模块在系统里的角色）

**功能描述**
（列出这个模块的功能点，对应规范书的功能要求）

**实现方案**
（用什么技术、什么算法、什么数据流）

**关键接口**
| 接口名 | 输入 | 输出 | 说明 |
|--------|------|------|------|
| ... | ... | ... | ... |

**对应需求**
本模块响应规范书 §X.X 的要求，覆盖功能点：……
```

每个模块至少 200-400 字，配一张模块交互图或数据流图。模块越多、每个写得越实，技术分越稳。

### 关键技术与创新点

挑 3-6 个亮点，每个亮点单独一小节：

- **技术名称**（如"基于 XX 的 XX 算法"）
- **应用场景**：在系统哪里用到
- **技术原理**：1-2 段说清原理，不要太学术
- **创新点 / 优势**：相比常规做法好在哪
- **效果验证**：性能数字、对比数据、专利/论文（如果有，用占位符 `{专利号}`）

注意：如果用户明确告知没有真实创新点，就用"应用层创新"代替"技术创新"——比如"针对 XX 行业场景的定制化方案"，避免编造专利。

### 性能与技术指标

最稳的写法是表格：

| 指标 | 招标方要求 | 本方案承诺 | 实测/设计值 |
|------|-----------|-----------|------------|
| 并发用户数 | ≥1000 | ≥2000 | 2400 |
| 响应时间 | ≤200ms | ≤150ms | 平均 120ms |
| 可用性 | ≥99.9% | ≥99.95% | 99.97% |

**承诺值要略优于招标要求，但别夸张。** 招标要求 99.9%，写 99.95% 或 99.99% 合理；写 99.9999% 评委会觉得不专业。

### 实施方案

包含：

- **实施方法论**：用什么标准（PMP / PRINCE2 / 敏捷 / 瀑布 / 自研方法），1-2 段说明
- **实施阶段**：通常分 5-6 阶段（启动 / 调研 / 设计 / 开发 / 测试 / 上线 / 验收 / 运维），每阶段写明工作内容、交付物、时间
- **甘特图**：用占位符 `{项目开始日期}` 起算，关键里程碑标出来
- **人员投入计划**：表格列出各阶段投入的角色和工时
- **风险与对策**：识别 5-8 个风险，每个写"风险描述—概率—影响—应对措施"

### 项目管理

按 PMBOK 或类似框架展开：组织管理、沟通管理、质量管理、进度管理、范围管理、配置管理、文档管理。每个管理过程写：

- 管理目标
- 管理流程（最好配流程图）
- 使用的工具（Jira / 禅道 / Git 等，用占位符或写通用工具名）
- 输出物

### 服务保障

涵盖培训、试运行、验收、售后、SLA：

- **培训方案**：分层级（管理层 / 业务层 / 技术层），列每层级的培训内容、时长、形式、教材
- **试运行方案**：试运行周期、监控指标、问题处理流程
- **验收方案**：分阶段验收（初验 / 终验），每阶段的验收标准、提交材料
- **售后**：质保期、响应时间分级（紧急 / 重要 / 一般）、响应承诺表
- **SLA**：可用性、响应时间、解决时间、违约责任

响应时间承诺表是评分常考点，必写：

| 故障级别 | 响应时间 | 解决时间 | 升级机制 |
|---------|---------|---------|---------|
| 紧急（系统宕机） | 15 分钟 | 4 小时 | 30 分钟无进展升级到经理 |
| 重要（功能异常） | 30 分钟 | 8 小时 | … |
| 一般（咨询/小 bug） | 2 小时 | 3 工作日 | … |

### 公司实力

- **公司简介**：成立时间、规模、主营、行业地位。**全部用占位符** `{公司名称}` `{成立时间}` `{员工人数}` `{年营收}`
- **资质证书**：列表展示。占位符 `{资质名1}` `{资质名2}`，并加一行"附件：详见附录 X 资质证书复印件"
- **类似项目案例**：3-5 个案例，每个案例写"项目名 / 客户 / 规模 / 时间 / 简介 / 业绩成果"。**全部占位符** `{案例项目名 X}` `{客户名 X}`
- **团队实力**：核心团队成员表（姓名 / 职位 / 学历 / 资历年限 / 持有证书），全占位符

### 投标承诺

通用模板，用占位符替换公司名：

> {公司名称}郑重承诺：
> 1. 完全响应招标文件全部要求，无任何偏离；
> 2. 严格按照投标文件履约；
> 3. 提供……（列具体承诺，5-8 条）

## 占位符规范

为方便最后批量替换，统一用花括号占位符，命名要可读：

- 公司：`{公司名称}` `{公司简称}` `{公司地址}` `{公司成立时间}` `{公司员工数}` `{公司年营收}`
- 项目：`{项目编号}` `{项目名称}` `{项目开始日期}` `{项目周期}`
- 产品：`{产品名}` `{产品版本}`
- 资质：`{资质名 1}` `{资质名 2}` …
- 案例：`{案例项目名 1}` `{案例客户名 1}` `{案例时间 1}` `{案例规模 1}` …
- 人员：`{项目经理姓名}` `{技术负责人姓名}` `{核心成员 1}` …
- 联系：`{联系人}` `{联系电话}` `{联系邮箱}`

输出工作稿时，给用户列出完整占位符与所需证据清单，便于补齐。输出正式终稿前必须重新扫描：只要仍有占位符或关键证据缺口，就不得标记为可投递终稿。

## 篇幅参考

中文技术标 100 分制下，常见篇幅 60-150 页。具体页数按评审表分值分配——10 分的章节一般 5-10 页，5 分的 2-5 页，1-2 分的 1-2 页。但页数不是越多越好：评委时间紧，废话太多反而扣印象分。**用图表和列表代替段落是涨页数又好看的稳妥做法。**
