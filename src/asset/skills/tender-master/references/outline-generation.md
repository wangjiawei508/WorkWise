# 目录生成（Outline Generation）

本文档展开 SKILL.md 第 4 步的细节。读这份文档时，假设你已经完成第 2 步（评审表清单）和第 3 步（规范书要求清单）。

## 标书的标准骨架

中文技术标书一般遵循这个粗框架，按需删减：

```
封面
投标函 / 投标声明
目录
1. 项目概述
   1.1 项目背景理解
   1.2 项目目标理解
   1.3 项目范围与边界
2. 需求分析与响应
   2.1 业务需求理解
   2.2 技术需求理解
   2.3 需求点对点应答表  ← 规范书逐条响应
3. 技术方案
   3.1 总体架构
   3.2 各功能模块设计
   3.3 关键技术与创新点
   3.4 技术指标与性能
4. 项目实施方案
   4.1 实施方法论
   4.2 项目计划与里程碑
   4.3 实施团队与人员配置
   4.4 风险识别与应对
5. 项目管理
   5.1 组织架构
   5.2 沟通机制
   5.3 质量管理
   5.4 进度管理
   5.5 文档管理
6. 服务保障
   6.1 培训方案
   6.2 试运行方案
   6.3 验收方案
   6.4 售后运维与质保
   6.5 应急响应与 SLA
7. 公司实力
   7.1 公司简介
   7.2 资质证书
   7.3 类似项目案例
   7.4 团队实力
8. 投标承诺
9. 附录
```

不要机械照抄。**评审表才是真正的骨架**，上面这个框架只是兜底，确保评审表里没明确提的"通用项"也覆盖到。

## 映射规则：评审项 → 章节

把评审表每一行映射到一个章节，遵循这些经验：

| 评审项关键词 | 目标章节 |
|-------------|---------|
| 项目理解 / 需求理解 / 背景认识 | 第 1 章 项目概述 |
| 总体方案 / 技术架构 / 系统设计 | 3.1 总体架构 |
| 功能模块 / 业务功能 / 模块设计 | 3.2 各功能模块（按模块拆子节） |
| 创新性 / 先进性 / 关键技术 | 3.3 关键技术与创新点 |
| 性能 / 容量 / 并发 / 响应时间 | 3.4 技术指标与性能 |
| 实施方案 / 实施步骤 / 进度计划 | 第 4 章 实施方案 |
| 项目管理 / 团队组织 / 沟通 | 第 5 章 项目管理 |
| 培训 / 验收 / 试运行 | 第 6 章 服务保障对应小节 |
| 售后 / 运维 / 质保 / SLA / 响应时间 | 6.4 / 6.5 |
| 公司资质 / 业绩 / 案例 | 第 7 章 公司实力 |
| 安全 / 保密 / 数据合规 | 单独提一节，通常放 3.x 或 6.x |

如果评审表里有招标方独创的评分项（比如"对 XX 行业的理解"、"本地化服务能力"），直接在对应位置加一节，标题就用评审项的原文最稳，让评委一眼对上号。

## 映射规则：规范书要求 → 章节

规范书的要求按内容归类：

- **功能性要求** → 3.2 各功能模块。每个模块单独成节，模块下面按"功能描述—实现方案—接口/数据"三段式
- **非功能性要求**（性能、可靠性、安全、可扩展性）→ 3.4 或独立成章
- **接口要求** → 3.2 模块下的"接口设计"小节，或单独"系统集成"章节
- **交付物清单** → 4.x 或附录里的"交付物清单表"
- **时间节点 / 工期** → 4.2 项目计划与里程碑
- **合规要求**（等保、密评、信创等）→ 安全章节 + 附录附上承诺函

带 ★/▲/【必须】/【重要】 标记的条款，**单独抽出来在标书显眼处响应一次**（通常做一张"关键要求点对点应答表"放在第 2.3 节），评委找起来不费劲。

## 输出格式

把目录用 markdown 列表给用户，每条三级标题后用 HTML 注释标出来源，方便用户核对：

```markdown
# 标书目录草案

## 封面
## 投标函
## 目录

## 1. 项目概述
### 1.1 项目背景理解        <!-- 规范书 §1.1 -->
### 1.2 项目目标理解        <!-- 评审表 1.1，5分 -->
### 1.3 项目范围与边界      <!-- 规范书 §1.3 -->

## 2. 需求分析与响应
### 2.1 业务需求理解        <!-- 评审表 1.2，5分 -->
### 2.2 技术需求理解
### 2.3 关键要求点对点应答表 <!-- 规范书 ★条款汇总 -->

## 3. 技术方案
### 3.1 总体架构设计        <!-- 评审表 2.1，10分 -->
### 3.2 功能模块设计
#### 3.2.1 数据采集模块     <!-- 评审表 2.2.1，5分；规范书 §2.1 -->
#### 3.2.2 数据处理模块     <!-- 评审表 2.2.2，5分；规范书 §2.2 -->
...

## 评审表覆盖检查
- 评审项 1.1（5分）→ §1.2 ✓
- 评审项 1.2（5分）→ §2.1 ✓
- 评审项 2.1（10分）→ §3.1 ✓
- ...
- 总计：100 分全部覆盖 ✓ / 漏 X 分 ✗
```

底部那张"覆盖检查"表很重要，帮用户一眼看清没漏评分项。

## 用户确认时常见反馈

- "把 XX 章拆得更细" → 拆三级/四级标题
- "顺序调一下，把技术方案放前面" → 按用户调
- "加一章 XX" → 加，并提示用户这一章对应哪个评审项（如果对应不上，提示用户这章可能没分但有助于整体观感）
- "太长了，砍掉一些" → 跟用户确认砍掉的部分是否对应低分评审项，高分项不能砍

确认无误再进入第 5 步。
