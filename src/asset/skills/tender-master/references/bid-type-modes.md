# 标的类型四模式判定与撰写规则（步骤①.5）

在提取需求和搭目录**之前**，必须把项目归入唯一一个主模式。模式是控制变量，不是展示标签，它决定需求台账、证据矩阵、目录结构、页数预算、撰写密度、表格设计与质检规则。

## 判定依据

从招标语言、采购标的、评分项、交付物、证据要求、验收方式六方面综合判定。若项目既含产品供货又含非平凡的软件/平台/服务实施，默认 `hybrid-goods-software-mode` 并分轨响应，不要把产品参数与平台方案混写。若招标语言与模式假设冲突，**一律服从招标**（固定响应表/页数上限/强制格式覆盖厚度默认值）。

## 四模式

| 模式 | 适用 | 核心材料 | 撰写厚度 |
| --- | --- | --- | --- |
| `goods-mode` 货物类 | 设备、硬件、标准产品、成品软件、耗材、仪器等现货 | 产品彩页、型号规格书、检测报告、厂家授权、证书、参数响应表、偏离表、质保、供货安装计划 | 最固定。优先参数、证据、追溯、偏离控制、供货安装、质保、验收。勿造臃肿通用方案章节 |
| `software-platform-mode` 软件平台类 | 定制开发、系统集成、数据中台、业务平台、AI 能力中心、运营平台、软件密集服务 | 需求分析、架构、功能设计、业务流、数据/接口/安全、实施、迁移、测试、演示、验收、运维 | 最厚。按功能、场景、数据对象、接口、流程、风险、记录、验收映射展开 |
| `hybrid-goods-software-mode` 软硬结合 | 硬件/标准品 + 平台开发/系统集成/部署/运维/服务 | 产品参数证据 + 软件平台方案 + 集成部署联调测试培训试运行验收记录 | 分三轨，各自可追溯 |
| `service-mode` 服务类 | 运维、维护、咨询、规划、培训、评估、数据治理、检测、监理 | 服务范围、人员、角色矩阵、SLA/KPI、流程、工具、记录、升级、报告、培训、交接、考核、验收 | 中到厚。聚焦人、流程、记录、KPI、风险、验收，勿套产品参数或软件架构 |

## 模式规则

- **货物类**：围绕 `参数响应表 + 产品证据矩阵 + 偏离表 + 厂家/材料追溯 + 供货安装调试方案 + 售后质保方案 + 验收交付清单`；正文支撑证据与履约，不替代参数证明。一份薄而完整的参数/证据响应，好过一长篇通用架构章节。
- **软件平台类**：激进使用厚标书规则。每个重要功能应说明业务场景、用户角色、流程、数据对象、接口关系、权限/安全、异常处理、测试、演示点、交付物、验收映射。
- **软硬结合**：建三套平行台账与目录——产品响应轨、软件平台方案轨、集成交付轨。加一张责任矩阵串联设备安装、软件部署、接口联调、系统测试、培训、试运行、终验。
- **服务类**：围绕服务成果写——服务目录、人员模型、排班/响应机制、SLA/KPI、工具与记录、问题升级、报告节奏、考核方法、知识转移、连续性、验收。勿强塞产品参数表或软件架构章节。
- **纯工程施工**：本 skill 可支持需求提取与评审，但施工组织设计、工程量清单、安全文明施工、施工进度需另配工程标流程。
