# 厚标书撰写规则（步骤⑤，软件平台/复杂混合类）

适用于标的额 3000 万以上、多系统平台项目、重大服务项目，或评分语言含"完全满足且优于项目需求"。按标的模式调整厚度。

## 核心规则

- **把招标源映射进正文**：把评分项、采购需求、技术参数、商务条款、交付物、验收要求连接到具体响应内容。避免"建立机制""加强管理"等抽象承诺，除非说清机制怎么运行、谁负责、何时执行、形成什么记录、记录如何支撑验收。
- **三级需求分解**：Level 1 需求域/平台/服务领域/管理主题 → Level 2 具体功能/服务任务/控制对象/交付组 → Level 3 最小响应单元（可写、可查、可举证、可验收）。以 Level 3 为正文、表格、验收映射的最小单位。
- **长度密度匹配复杂度**：不要把重大软件平台/服务/混合方案章节停留在 5000–10000 字的概览级。用专项控制内容、流程细则、检查表、台账模板、风险预防措施、阶段门禁、交付记录、验收映射表展开。
- **最小小节保持实质**：每个最小正式正文小节通常 ≥3-5 段真实内容；货物类参数响应、固定表单、合规声明、招标指定短响应可更少。不要用单段或薄表格替代需评分的响应。
- **按页数规划全案**：招标无硬性页数上限时，软件平台/复杂混合以 `合同额每 1 万元 ≈ 1 页` 为基线，再按系统复杂度、评分项数量、集成难度、数据迁移、演示、验收义务上调。重大软件平台/复杂混合技术商务方案一般不低于 300-500 页；货物类由参数/证据完整度决定；服务类按 SLA、人员、记录、考核规模。
- **降 AI 味**：避免"全过程、全链路、全角色、全闭环"式对仗口号。用具体项目场景、业务对象、数据流、接口联调、角色协作、质量记录、问题处理、验收材料、采购方协同细节。
- **首选结构 `机制 + 场景 + 表单 + 输出成果`**：
  - 机制：管理方法与责任角色
  - 场景：如何应用于本项目业务/技术边界
  - 表单：固化流程的台账、检查表、会议纪要、测试记录、迁移记录、验收映射
  - 输出成果：支撑评审、实施、交付、验收的材料
- **项目理解章节**：避免宽泛政策套话。按 `政策背景 → 采购人定位 → 建设目标 → 业务现状理解 → 信息化现状判断 → 痛点归纳 → 建设必要性 → 价值链/业务链分析 → 响应策略` 构建。先用招标语言，再谨慎推断业务含义，勿臆造采购方内部数据。
- **平台软件项目写专项方案**而非通用软件工程：覆盖生产/流通/交易服务平台、数据中台、AI 能力中心、电子合同、质量追溯、接口联调、数据迁移、主数据管理、业务连续性、演示功能等相关边界。每个专项说明质量风险、控制重点、检查方法、输出结果。
- **表格要可评审而非装饰**：只在支撑评分或实施控制时用表——质量指标表、阶段门禁表、岗位职责表、问题分级表、接口联调检查表、数据质量检查表、风险预防表、交付文档清单、验收映射表。每张表应帮助证明"完全满足且优于项目需求"。
- **评分章节先读目标分与评分语言**：当评委期望"优于"，通过更细的流程控制、更全的风险预防、更清晰的验收联动、更强的专项措施提供证据，而非复述招标原文。
- **通用章节本地化**（质量管理、组织实施、验收交付、运维服务）：结合本项目。如质量管理章节应含质量目标、指标、职责、问题处理、评估、整改、奖惩、生命周期质量门禁、平台专项质量控制、数据中台质量控制、接口联调控制、数据质量控制、试运行控制、质量会议、流程表单、风险预防、文档一致性控制、采购方协同、验收跟踪。
- **不留正式章节草稿痕迹**：`本章自查`、`待确认`、`TODO`、`公司名称`、`项目名称：填写`、`如有证据再补` 等。人员、证书、业绩、软著、授权、厂家证据不可得时，引用正式章节/附件，如`详见本投标文件《项目组织实施方案》及《本项目管理人员及服务人员名单》`。

## 输出标准

- 每个关键论述可追溯到招标原文、用户提供的企业证据或明确假设；
- 需求台账、评分映射、证据矩阵、评审结论用表格；
- 用可评估语言：`满足/优于/响应/提供` 绑定可度量招标条款；
- 抛光前先标注 `否决项`、缺失证据、不可能的交期、矛盾条款、格式约束；
- 不伪造截图、证书、案例、授权函、签章状态、第三方承诺。
