# 地保监测综合 Skill

本 Skill 面向“轨道交通控制保护区第三方监测/监护”的完整工作链：前期资料收集、监测方案、内审专审、测点埋设验收、初始值采集审批、现场实施、日报周报月报、预警报警消警、控制网复测、项目总结。使用时要把样例资料当作“格式和逻辑参考”，不能把样例项目名、单位名、合同号、仪器号、点号数量直接带入新项目。

## 总原则

1. **先建项目资料卡，再写任何成果。** 若用户没有提供完整信息，先从文件名、报表封面、`info` 表、已有方案中提取；仍缺失时保留 `{{占位符}}`，不要从参考资料臆造。
2. **先判断阶段，再加载资料。** 方案编制、周期出报、预警处置、评审回复的关注点不同，不要用一个模板硬套所有任务。
3. **先判断监测对象。** 地下盾构/车站和高架桥墩/桥面的监测项差异很大；高架项目不能默认写“隧道收敛”，应改为桥墩倾斜、桥墩沉降、桥墩位移、桥面道床/结构沉降等。
4. **先确认阈值来源。** 报警值优先级为：本项目监测方案/安评报告/专家意见/运营单位指令 > 用户给定阈值 > 参考资料默认经验值。高架和桥墩类没有项目阈值时必须标注“待确认阈值”。
5. **报表必须可审计。** 每个数值要能回到原始 Excel、CSV、日报、周报、月报、控制网报告或预警记录；不得只写“总体稳定”这类无依据结论。

## RAILWISE-OS Native 模式

当本 Skill 由 RAILWISE-OS / RAILWISE-CLI 调用，且工作区已提供 `input/dibao-context.json`、`input/locked-report-preview.json`、`input/report-blueprint.json` 时，必须优先遵守 OS Native 数据源策略：

1. **OS 数据优先。** 正常出报时只使用 RAILWISE-OS 已入库并锁定的数据，不要再追问用户提供老平台 URL、账号、密码、Cookie 或接口参数。
2. **AI 不改数值。** 项目、周期、测点、阈值、预警、统计值、全站仪成果、静力水准成果均以 `locked-report-preview.json` 为准；AI 只写工况说明、进度摘要、风险解释、结论建议和待补资料。
3. **数据补救才追问外部来源。** 只有 OS 明确提示本周期无数据、数据晚到、测点未绑定、质量待复核、锁定失败，或用户主动要求历史补跑时，才进入“等待同步 / 补拉老平台 / 上传成果文件 / 生成缺测说明”的补救流程。
4. **全站仪成果口径。** 全站仪高频报表只能使用平差后成果或平台已经作为成果展示的数据；不得用首页最新动态、未完成观测批次、原始过程坐标表直接生成正式报表。
5. **静力水准取样口径。** 静力水准补拉或出报必须明确 `statDate`、`endDate` 和 `sampMinutes` 取样间隔；正式结论使用 `curOffset` / `totalOffset` 沉降成果，不把原始液位读数直接当沉降成果。
6. **出报频率按项目要求。** 15 分钟、2 小时、4 小时或自定义频率都应由项目要求或用户选择决定；不要把 4 小时写成所有项目的固定规则。

## 一、资料收集与立项阶段

适用表达：“新接了个项目”“准备写方案”“资料还没齐”“帮我列资料清单”“工作量/报价怎么算”。

执行顺序：

1. 用 `assets/project-brief-template.md` 建立资料卡。
2. 按 `references/phase0-intake.md` 一次性索取资料，不要零散追问：
   - 外部工程资料：总平面图、基坑/盾构/道路/桥梁施工图、施工组织设计、工筹横道图。
   - 轨道资料：线路、车站、区间、桥墩/桥面结构图，既有沉降资料，保护区范围。
   - 勘察和环境资料：地勘、水文、管线、周边建筑物、道路交通和重载车辆路径。
   - 安评资料：安全影响评估报告、审查意见、控制指标。
   - 合同和管理资料：委托单、合同、运营单位要求、报表格式。
3. 判断监测对象：
   - 地下区间/车站：沉降、水平位移、收敛、巡视，必要时静力水准、振动、轨道几何。
   - 高架/桥梁/出入场线：桥墩沉降、桥墩水平位移、桥墩倾斜、盖梁/梁端/承台/桥面沉降、巡视。
4. 估算监测等级、测点数量、监测频率、人工/自动化工作量，详见 `references/scoping-and-pricing.md`。
5. 输出资料缺口清单和下一步动作，不要在资料不足时假装可以定稿。

## 二、监测方案编制阶段

适用表达：“写监测方案”“帮我起草方案”“重难点怎么写”“监测范围/频率/报警值怎么定”。

推荐起草顺序：

1. **先写监测范围和影响关系。** 明确保护区范围、线路/车站/区间/桥墩编号、最近距离、竖向关系、风险源。
2. **再写施工筹划。** 从工筹横道图提取阶段：围护、开挖、支撑、盾构到达/穿越、顶进、桩基、桥梁拆改、回填、收尾等。
3. **再写监测项目和测点布设。** 区分自动化、人工复核、巡视、控制网；点号规则要与图纸和报表一致。
4. **再写监测频率分阶段表。** 普通阶段、关键工序、预警报警阶段分别列频率，不要只写一个笼统频率。
5. **单独写重难点及措施。** 用 `references/key-difficulties.md` 的“诊断—推演—生成”方法，不要写泛泛的“加强监测”。
6. **再写报警值和预消警流程。** 引用安评和运营单位要求，分监测项列预警值、报警值、控制值、速率/趋势报警条件。
7. **最后补工程概况、编制依据、控制网、监测方法、组织机构、质量安全、应急预案、资料提交。**

模板选择：

- 运营线路或运营单位要求完整格式：优先使用模板 A，详 `references/toc-template-A.md`。
- 建设期线路、小型项目或设计院简版：使用模板 B，详 `references/toc-template-B.md`。
- 默认使用 `assets/plan-template.md` 起稿，再按项目删改。

高架/桥梁项目特别规则：

- 必须加载 `references/elevated-line.md`。
- 不要默认写盾构区间“环片收敛”；除非确有隧道结构，否则改为桥墩倾斜、墩顶/墩底沉降、盖梁/梁端/桥面监测。
- 控制网首级点优先布设在桥外稳定区域，说明登高、接触网、运营窗口、视线遮挡、安全防护等约束。
- 报警值必须来自本项目方案/安评/专家意见；没有就列“待确认”，不要套地下隧道阈值。

方案质量检查：

- 章节编号和目录一致。
- 图表有编号和题注。
- 监测范围、监测项目、点位图、工作量表、频率表、报警值表互相一致。
- 平面控制网和高程控制网分别写清基准点、工作基点、监测点。
- 预警、报警、控制值不能混成一列。
- 施工阶段的频率要分阶段，不要一张表覆盖所有风险阶段。

## 三、内审和专家评审阶段

适用表达：“明天内审”“帮我准备内审材料”“专家意见怎么回复”“改审查意见”。

内审准备：

1. 读 `references/phase2-internal-review.md`。
2. 用 `references/review-checklist.md` 自查方案，重点查：监测范围、测点数量、频率、报警值、重难点、控制网、应急流程、附图。
3. 形成内审议程、参会人员、问题清单和修改责任。
4. 内审后用 `references/review-response-template.md` 写逐条回复，回复必须对应页码或章节。

专家评审回复：

1. 逐条拆分专家意见，不合并不同问题。
2. 每条回复写“采纳情况 + 修改位置 + 修改后内容摘要”。
3. 对无法完全采纳的意见要说明原因和替代措施。
4. 回复表中的页码、章节号必须和修订版方案一致。

## 四、初始值采集与开测审批阶段

适用表达：“做初始值”“写初始值报告”“测点埋设记录”“测点验收”“控制网第 1 期联测”“开测前资料”。

初始值是后续累计变量和报警判定的基准，必须作为独立阶段处理。没有审批确认的初始值，不应直接进入正式日报。

执行顺序：

1. 先读 `references/initial-and-summary-workflow.md`。
2. 确认监测方案、点位图、监测项目、报警值、频率已经明确。
3. 整理测点埋设记录：项目名称、监测单位、监测项目、测点数量、测点编号、对应监测对象、埋设时间、环号/桥墩号/位置、埋设说明、点位示意图、现场照片、验收签字。
4. 整理测点验收记录：人员到位、仪器检定、测点数量、点号范围、点位状态、监测/监护/运营或地保相关方意见。
5. 整理平面控制网第 1 期联测：坐标系统、网形、首级控制点、工作基点、观测仪器、技术指标、成果是否作为后续监测起算数据。
6. 整理高程控制网第 1 期联测：高程基准点、工作基点、水准路线、观测顺序、闭合差/中误差、使用高程、成果沿用或更新结论。
7. 整理监测初始值审批表：人工监测初始值和自动化监测初始值分章；地下项目按道床沉降、结构沉降、远端沉降、收敛、水平位移分节；高架项目按桥墩沉降、桥墩位移、桥墩倾斜、桥面沉降、桥面道床/结构沉降分节。
8. 使用 `assets/initial-value-report-template.md` 形成初始值报告或审批资料。

初始值资料交付前检查：

- 测点编号、数量、监测项目与方案和布点图一致。
- 人工和自动化初始值分开列明。
- 沉降、位移、收敛、倾斜的单位清楚。
- 初始值日期早于正式监测日报日期。
- 平面/高程控制网基准点、工作基点、使用成果明确。
- 缺点、漏点、异常点必须说明原因和补采计划。
- 审批表、验收表、原始记录、照片、点位图、仪器检定资料齐全。

## 五、现场实施和数据整理阶段

适用表达：“整理日报 Excel”“统计监测数据”“判定报警”“从这些表里提取最大值”。

数据处理顺序：

1. 先读 `references/data-processing.md`。
2. Excel 优先找 `info`、`监测日期`、`适配设置页`、`Z1`、`Z2`、`R1`、`R2`、`统计表`、`统计分析` 等辅助表。
3. 提取项目元数据：项目名、单位、日期、期号、天气、施工工况、报警状态。
4. 提取监测数据：监测项目、方法、部位、点号、本次变量、速率、累计变量、阈值。
5. 归一化为 `assets/data-input-template.csv` 的字段。
6. 有 CSV 数据时可运行：

```bash
python scripts/evaluate_alarms.py input.csv
```

7. 报表统计时分别计算：
   - 本次/本期最大变化量点号和值。
   - 该点对应累计值。
   - 累计最大变化量点号和值。
   - 该点对应本次/本期变化量。
   - 状态：正常、预警、报警、红色预警、待确认阈值。

数据易错点：

- “本次最大”和“累计最大”通常不是同一个点，不能合并。
- 自动化监测和人工监测的日期范围可能不同，周报/月报必须分别说明。
- 公式表可能有隐藏表、缓存值、外部图表或断裂引用；不要只看封面。
- 速率报警和累计报警是两条线；`是否数据报警` 与 `是否综合报警` 不可混填。
- 负号方向必须与本项目说明一致。

## 六、穿越工况自动化全站仪高频报表

适用表达：“穿越期间 15 分钟出一次报表”“盾构/顶管穿越 2 小时快报”“4 小时自动化报表”“先做全站仪的高频报表”。

3. 明确监测方式：本版按自动化全站仪/测量机器人处理；静力水准另行扩展，不混写。
4. 版式参考样例：微信群发送版优先做 A3 横向单页 Excel 快报，横版通栏排版，密集黑色表格线；“施工工况”“施工进度”“各项数据分析”“结论与评价”必须横向合并成一栏，不要做左右两套重复分栏。“工况及影响范围示意图”应独立成通栏图片区，下面左、右各放一张图；图片必须按图片区裁切填充，不留白、不越界。“平台最新动态”放在图片区下方，做左右两组 4 列信息表。底部放本次最大、累计最大、备注、结论与评价，不保留空边框行。
   - 地下盾构穿越既有地下区间且用户提供通途西路类工程师模板时，报表不采用桥墩倾斜模板；应按 `references/crossing-underground-shield-combined-report.md` 生成：`前台` 双列上/下行线快报、`前台 (静力水准)` 结构沉降页、`Z1 全 (2)` 和 `Z1 （静力水准)` 数据页、累计曲线图和必要隐藏辅助页。
   - 正式留档/完整报表必须同时输出 5 页版本：封面、全站仪快报、桥墩沉降明细、桥墩水平位移明细、桥墩倾斜明细。完整报表使用 `assets/完整报表模板.xlsx`，导出 PDF 时隐藏 `适配设置页`，不把设置页算入页数。
   - 以后穿越期高频报表默认按项目指定频率生成两套：`{{项目全称}}_{{YYYYMMDD}}_{{小时}}点_{{频率标签}}快报.xlsx/pdf` 和 `{{项目全称}}_{{YYYYMMDD}}_{{小时}}点_{{频率标签}}完整报表.xlsx/pdf`。
5. 高架桥墩有倾斜监测要求时，必须计算倾斜率。若平台或人工成果已有 `MCQX/倾斜率` 字段，优先直接采用该成果；否则按同一桥墩上下测点成对计算，例如 S12/X12、S13/X13。当前项目按工程师公式：单次倾斜 `((本次S点X-上次S点X)-(本次X点X-上次X点X))/上下点初始三维距离×1000`；累计倾斜 `((本次S点X-初始S点X)-(本次X点X-初始X点X))/上下点初始三维距离×1000`。位移单位为 m，结果单位为 ‰；正值为北方向偏移，负值为南方向偏移。上下点初始三维距离用平台自动化初值 `X/Y/H` 计算，缺失时写“初始三维距离待确认”，不得硬算。
6. 点位角色必须先判别：`CZ` 前缀为测站，`JD` 前缀为测量基点/基准点，不作为监测点，不参与监测点数量、本次最大、累计最大和数据表统计；可在平台摘要中说明“测站/测量基点”数量。
7. 点名映射必须人工确认：若平台点号为 `S/X`、报表点号为 `MCC/MCQX` 或其他体系，先按编号、布点图和初始值报告建立映射表，经用户/工程师确认后再生成正式报表。未确认时只输出草稿或数据核对表。
   - 本项目报表可见点名不得出现 `S12/X12/S13/X13` 等平台原始点名。沉降映射为 `MCC`，水平位移映射为 `MCW`，且二者均按下部 `X` 点填列和统计最大值；倾斜映射为 `MCQX`，按对应 `S/X` 上下点配对计算。
8. 自动化全站仪快报的变形成果优先采用平台“变形监测平差数据”的平差后坐标和平差批次。当前项目沉降、水平位移和倾斜按工程师表坐标公式计算；平台累计 `dX/dY/dH` 用于反推并校核自动化初值，不直接用人工初始值报告起算。人工初始值报告只可作为项目实施单位、项目信息等文字元数据来源；报表起算来源写“全站仪自动化平台初始值/平差变形成果”，不写“人工初始值”或“平台未给出具体时间”。
   - 若工程师对个别点位/监测项做人工复核修正，不要直接改 Excel 成果；应写入 `manual_metric_overrides.json` 一类的覆盖表，至少包含报表时间、平台点号、报表点号、监测项、方向、本次变化值、累计变化值和原因，再通过 `--manual-overrides` 生成，使平台动态、主表、明细公式、最大值、曲线图和留痕一致。
9. 底部编制信息写项目实施单位，优先从初始值报告中的监测单位/项目实施单位提取；报表类型统一写 `轨道交通控制保护区监测快报`。
10. 数据表按监测对象分组，常见列为道床沉降、结构沉降、水平位移、水平收敛、桥墩倾斜率；没有的监测项删除，不能保留空壳。
11. 黄色宜表示当前穿越正投影或重点影响区段，绿色只标记主数据区本次/累计绝对值最大但未报警的数据；“本次最大/累计最大”汇总行只写点号和值，不填绿色。红色只用于达到本项目预警/报警/控制条件的数据。
12. 快报页沉降、水平位移按 0.1mm 显示，桥墩倾斜按 0.01‰ 显示；完整报表的桥墩倾斜明细页坐标和倾斜计算值保留 5 位小数，并生成桥墩沉降、桥墩水平位移、桥墩倾斜三张曲线图。源 CSV 保留原始计算值用于追溯。
13. 缺测、遮挡、仪器离线、反射片异常不得填 0；应保留 `/` 或空值并在备注写明原因和复测/补测安排。
14. 输出文件名必须使用业务命名并带名义报表日期，不要使用技术型前缀、接口批次时间戳或内部口径词。推荐格式：`{{项目全称}}_{{YYYYMMDD}}_{{名义出报小时}}点_{{频率标签}}快报.xlsx`，例如 `示例轨道交通控制保护区监测项目_20260605_12点_2小时快报.xlsx`；PDF 同名，仅扩展名改为 `.pdf`。其中日期和小时均按报表名义截止时间取值，不按实际触发时间或平台批次时间取值。
15. 交付前必须清理系统缓存和临时文件：运行 `scripts/clean_report_artifacts.py "{{输出目录}}"`，删除 `.DS_Store`、`__MACOSX/`、`._*`、`Thumbs.db`、`desktop.ini`、`~$*`、`__pycache__/`、`*.pyc` 等。打包 zip 或发送给 Windows 用户前，加 `--check-archives` 检查压缩包和 Office 文件内部是否夹带 macOS 隐藏条目。
16. Excel/PDF 版式一致性以最终 PDF 为准。Mac Excel、Windows Excel、WPS、LibreOffice 的字体度量、默认打印机、图片锚点和分页可能不同；正式留档应固定导出环境，优先用同一版本 LibreOffice headless 或同一台 Windows Excel 导出。模板必须固定 A3 横向、打印区域、缩放到 1 页宽/1 页高、行高列宽和字体；Windows 缺少 `SimSun/宋体` 时应先安装或改用项目统一字体。

高频报表交付前检查：

- 报告时间和自动化数据时间一致，不能用未来数据补当前快报。
- 本次最大和累计最大分别统计，且点号、数值、监测项、分区能追溯到源表。
- 报警阈值来源明确；若项目阈值缺失，状态写“待确认阈值”。

- 输出目录无 `.DS_Store`、`__MACOSX`、`._*`、`~$*.xlsx` 等缓存文件；PDF 在最终交付环境打开后分页、图片、图表、边框无明显错位。

```bash
  --project-id "{{华桓项目ID}}" \
  --project-name "{{项目全称}}" \
  --report-cadence "{{15min/2h/4h/custom}}" \
  --report-cutoff-time "{{本次报表名义截止时间，可省略用平台最新}}" \
  --previous-time "{{上期参考时间，补跑时建议显式填写}}" \
  --samp-minutes 60 \
  --output-dir "平台数据输出"
```

脚本会调用华桓 `TESTAPI`：`finddateById` 获取项目最新时间，`findSZByIdAndDate` 以 `type=2` 获取静力水准/沉降的本期变化和累计变化，并输出原始 JSON、规范化 CSV 和 Markdown 摘要。网页登录只用于查找用户名可见项目、确认项目 ID 或取 Cookie；正式数据计算优先走接口文件，不用页面截图里的值硬抄。

华桓静力水准数据口径：

- `findSZByIdAndDate` 返回的 `curOffset` 为本期变化，`totalOffset` 为累计变化；报表本次/累计优先直接采用该接口成果。
- `curValue`、`refValue` 为本期/参考测值，可放入完整报表明细页；`curOriginalValue`、`refOriginalValue` 只作追溯。
- 单位缺失时先按 mm 处理，但必须在源摘要中写明“接口 unit 为空，按项目静力水准沉降 mm 口径填报”。
- 正负号方向以项目报表备注为准；没有项目说明时，沉降常用“+ 为隆起，- 为下沉”，但应标注待确认。
- `status` 不等于正常或数值缺失时，不得填 0；写 `/` 或空值并备注离线、缺测、异常或待复核。
- 文件名仍按业务命名：`{{项目全称}}_{{YYYYMMDD}}_{{小时}}点_{{频率标签}}静力水准沉降快报.xlsx/pdf`。

## 七、监测日报

适用表达：“写日报”“出第 N 期日报”“把今天数据做成监测日报表”。

使用 `assets/daily-report-template.md`。日报通常包含：

1. 封面：项目名、日报编号、日期、编写/校核/审核、监测单位。
2. 监测分析报告：
   - 天气和日期。
   - 当日施工工况。
   - 监测项目数据统计表。
   - 正负号说明。
3. 前台/综合分析页：
   - 地下项目按上行线、下行线或车站/区间分组。
   - 高架项目按桥下、桥墩、桥面、桥面道床/结构等分组。
4. 单项监测日报表：
   - 每个监测项单独成表。
   - 页眉包括项目名、单位、合同号、日报编号、报警字段、仪器、日期、间隔、工况、气象。
   - 数据表包括点号、本次变量、速率、累计变量、初始/上次/本次值或坐标。
5. 结论与评价：
   - 正常时写最大值和趋势。
   - 预警/报警时写点号、数值、阈值、措施。

日报交付前检查：

- 是否还有旧项目名、旧单位名、旧日期。
- 日报编号和日期是否一致。
- 是否写明自动化/人工实际监测间隔。
- 每张子表 `是否综合报警`、`是否数据报警` 是否正确。
- 正负号备注是否在汇总表下方保留。

## 八、监测周报

适用表达：“写周报”“本周工作量统计”“下周计划”“自动化人工对比”。

使用 `assets/weekly-report-template.md`。周报通常包含：

1. 封面：项目名、周报编号、统计期、监测单位、报告日期。
2. 本周监测工作量统计：
   - 日常监测、巡视巡查、会议、联系单、本周完成、下周计划。
3. 外部工程/现场施工工况：
   - 本周保护区内施工进展和下周风险工序。
4. 监测数据汇总：
   - 自动化和人工可分行列出。
   - 地下项目常见道床沉降、结构沉降、水平位移、收敛、远端沉降。
   - 高架项目常见桥墩沉降、桥墩位移、桥墩倾斜、桥面沉降。
5. 存在问题及建议：
   - 预警/报警。
   - 数据异常或复测。
   - 施工建议。
6. 下周工作计划。

周报特别注意：

- 工作量单位要沿用源资料：`4h/次`、`6次/天`、`点次`、`1次/周` 等不能乱改。
- 自动化与人工对比只在同类监测项和可比区间存在时写。
- 若差值异常，说明复测或核查计划，不要简单写“正常”。

## 九、监测月报

适用表达：“写月报”“本月监测总结”“控制网复测”“预消警情况”。

使用 `assets/monthly-report-template.md`。月报通常包含：

1. 工程概况及进度：
   - 工程概述、位置关系、轨道交通结构概况、施工工筹、本月进度。
2. 本月工作开展情况：
   - 监测工作量、巡视、会议、培训、联系单、项目管理。
3. 控制网复测情况：
   - 高程控制网和平面控制网分开写。
   - 写明本期是否复测、是否沿用原成果或更新成果。
4. 监测数据汇总及分析：
   - 汇总表。
   - 曲线图分析。
   - 最大值、趋势、施工影响判断。
5. 监测预、消警情况：
   - 无事件时明确写“本月未发生监测预警、报警及消警事项”。
   - 有事件时列日期、级别、点号、数值、措施、当前状态。
6. 存在问题及建议。
7. 巡视。
8. 附件：开工时间确认单、监测报表、控制网成果、预警资料、照片、会议纪要等。

月报特别注意：

- 不要默认写“工作基点稳定，沿用原成果”，除非控制网报告支持。
- 曲线图缺失时要列清需要哪些曲线，不要编造图号内容。
- 统计期要区分自然月、合同月和实际报表期。
- 月报里的工程概况可以继承上一期，但施工进度、工作量、数据和预消警必须更新。

## 十、预警、报警、红色预警、消警

适用表达：“出预警单”“报警怎么写”“红色预警汇报”“消警资料”“数据说明”。

先读 `references/warning-workflow.md`，再用 `assets/warning-notice-template.md`。

预警资料必须写清：

1. 级别：预警、报警、红色预警、消警。
2. 部位：车站、区间、上/下行、桥墩编号、桥面区域、施工影响区。
3. 监测项目：道床沉降、结构沉降、水平位移、收敛、桥墩倾斜等。
4. 点号和数据：本次变量、速率、累计值、阈值、触发时间。
5. 施工工况：触发时段正在进行的关键工序。
6. 原因分析：结合趋势、工况、复测和历史曲线，证据不足时用审慎措辞。
7. 处置措施：通知、加密监测、复测、巡视、施工参数核查、专题会或专家评审。
8. 附件：曲线、日报页、原始数据、照片、复测记录、会议纪要。

消警资料必须有证据：

- 数据稳定达到要求的持续时间。
- 无继续发展趋势。
- 复测或人工核查结果。
- 风险工序完成或风险已受控。
- 运营/建设/施工相关方确认流程。

## 十一、项目总结报告

适用表达：“写项目总结”“监测项目总结报告”“停测总结”“完工总结”“监测监护总结”。

项目总结是整个监测服务的闭合文件，不是月报简单合订。它要说明监测做了什么、施工期间数据如何变化、预警是否闭合、是否满足停测/结束条件、后续有什么建议。

执行顺序：

1. 先读 `references/initial-and-summary-workflow.md` 的项目总结部分。
2. 收集监测方案、安评报告、评审意见和回复、初始值报告、测点验收记录、控制网联测和复测报告、日报、周报、月报、监护记录、预警/报警/消警资料、巡视记录、现状普查、施工节点、停测申请或停测确认、人员设备和工作量统计。
3. 使用 `assets/project-summary-template.md` 起稿。
4. 总结内容至少覆盖：目的及编制依据、项目概况、投入人员及设备、监测监护范围、周期及停测依据、工作量、监测内容/精度/频率、控制网建立与复测、信息反馈和预警处置、成果分析、监护巡查、安全生产、结论与建议。
5. 成果分析按“总体 + 分项 + 阶段”写：总体变形情况、各监测项最大累计值和最大速率、施工阶段变化规律、预警事件及闭合情况。
6. 结论必须说明轨道交通结构变形是否可控、是否发生预警/报警/红色预警、控制网是否稳定、是否满足停测或结束监测条件、后续是否需要继续观察或保留测点。

项目总结交付前检查：

- 监测周期、施工节点、停测依据一致。
- 工作量统计与日报、周报、月报数量一致。
- 初始值、控制网、日报、月报、预警资料能互相对应。
- 最大值、曲线和文字结论一致。
- 预警/报警/消警事件完整闭合。
- 附件完整，图表编号连续。

## 十二、监测项和阈值使用规则

默认分类见 `references/monitoring-catalog.md`，但必须服从本项目资料。

常见监测项：

- 地下区间/车站：道床沉降、结构沉降、水平位移、水平收敛、竖向收敛、远端沉降、巡视。
- 自动化：测量机器人、静力水准、自动化位移/沉降。
- 人工：水准测量、全站仪/自由设站、人工复核。
- 高架/桥梁：桥墩沉降、桥墩水平位移、桥墩倾斜、桥下沉降/位移、桥面沉降、桥面道床沉降、桥面结构沉降。
- 控制网：高程控制网、平面控制网、工作基点复测。

默认正负号：

- 沉降类：正值为上抬，负值为下沉。
- 水平位移：必须写清本项目正方向，常见为“正值向基坑侧偏移，负值反向偏移”。
- 收敛：按本项目模板定义张开/收缩方向。
- 桥墩倾斜：写明方向或分量，不要只写绝对值。

默认报警层级：

- 正常：未达预警值。
- 预警：达到预警值，未达报警值。
- 报警：达到报警值，或触发速率/趋势报警。
- 红色预警/超控制值：达到或超过控制值。
- 待确认阈值：当前项目阈值缺失，尤其高架、桥墩、专家调整值场景。

## 十四、最终质量检查清单

交付前逐项检查：

- 项目名、单位名、合同号、日期、期号全部属于当前项目，没有样例残留。
- 所有 `{{占位符}}` 已填完；无法填的已明确说明缺资料。
- 监测对象与监测项匹配，地下/高架没有混用。
- 初始值采集、测点验收、控制网第 1 期联测在正式监测前完成；若缺失，已明确缺口和风险。
- 阈值来源明确，预警值、报警值、控制值分列。
- 自动化和人工监测日期范围分别写清。
- “本次/本期最大”和“累计最大”分别统计。
- 穿越高频报表的出报间隔、上次/本次监测时间、统计时段、施工位置完整。
- `是否综合报警` 与 `是否数据报警` 分开判定。
- 正负号说明完整。
- 施工工况与统计期一致。
- 控制网复测结论有依据。
- 周报/月报的工作量单位和次数与源资料一致。
- 预警/消警资料有点号、数值、阈值、时间、措施和附件。
- 签字栏完整：测量、制表、复核、编制、校核、审核等按模板要求保留。
- 附件清单完整，尤其月报中的开工时间确认单、监测报表、控制网成果、预警资料。
- 项目总结中的监测周期、工作量、预警闭合、停测依据和结论建议完整。

## 十六、典型使用示例

### 写监测方案

用户说：“新接了一个基坑项目，帮我写地保监测方案。”

执行：

1. 建资料卡，确认线路、结构、施工类型、安评和工筹是否齐全。
2. 读 `phase0-intake.md` 补资料清单。

### 出日报

用户说：“帮我出第 12 期日报，Excel 在这个文件夹。”

执行：

1. 识别日报 Excel，先读 `info`、`监测日期`、`Z1/Z2/R1/R2`、`统计表`。
2. 提取日期、期号、天气、施工工况、监测数据。
3. 计算本次最大、累计最大、报警状态。

### 做初始值报告
