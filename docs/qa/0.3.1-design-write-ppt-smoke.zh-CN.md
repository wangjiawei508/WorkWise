## 真实应用测试环境阻断

测试机：

```text
macOS 26.5.2 (25F84)
Apple Silicon arm64
```

候选应用完成打包且 `codesign --verify --deep --strict` 通过，但 macOS 在启动新生成的本地临时签名应用时直接发送 `SIGKILL`。同机 Codex/ChatGPT 自带的 Computer Use Node 运行时也出现相同系统策略终止：

```text
/Applications/ChatGPT.app/Contents/Resources/cua_node/bin/node --version
status: SIGKILL

ELECTRON_RUN_AS_NODE=1 \
  dist/mac-arm64/WorkWise.app/Contents/MacOS/WorkWise \
  -p process.versions.modules
status: SIGKILL
```

系统仍能运行此前安装的 WorkWise 0.3.0，其 Electron ABI 为 148。新候选包内 `better-sqlite3` 也为 ABI 148，但从旧应用进程加载时被系统拒绝，错误为 `library load disallowed by system policy`。这说明阻断发生在本机新临时签名身份与动态库策略，而不是 ASAR 缺文件或 ABI 构建错误。

为避免破坏用户正在使用的 ChatGPT/Codex 与 WorkWise，不修改 `/Applications/ChatGPT.app` 的签名、不覆盖现有 WorkWise、不用 AppleScript 或其他自动化绕过 Computer Use。

## 结论与发布门禁

源码、集成测试、生产构建、候选 DMG 和包内容检查均通过。当前机器无法完成可交互的候选应用 Computer Use 冒烟，因此本记录属于 OpenSpec 允许的“具体环境阻断 + 可复现回退测试”，不能写成真实界面已通过。

下一步应在干净的 macOS runner 或测试机上使用正式 Developer ID 签名候选包，完成以下真实操作后再允许发布：

1. 新建 Design 文档，添加文字、图形和图片，组合后重启恢复。
2. 用 Design 助手修改当前画板并验证撤销/重做、冲突提示和保存状态。
3. 导入真实 PPTX，检查保真警告；导出真实 PPTX 并用 PowerPoint/WPS 打开。
4. 下载 PNG/SVG，检查文件非空、尺寸和图片资源。
5. 插入 Write，检查相对引用、预览和失败回滚。
6. 检查 PPTX、PNG、SVG 成果卡的“打开、另存为、显示位置”。

在这些操作完成前，0.3.1 保持候选状态，不创建版本标签和 GitHub Release。
